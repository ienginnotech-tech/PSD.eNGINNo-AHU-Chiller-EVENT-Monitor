อัปเดตจากเวอร์ชันก่อนหน้า: เพิ่มระบบ **Chiller Plant** เต็มรูปแบบ (Evaporator Loop, Condenser Loop, SCHP แยก Phase)
และเปลี่ยนฐานข้อมูล AHU ทั้งหมดให้เป็น **ข้อมูลจริงของ BITEC** จาก Machine List และเอกสารพื้นที่/ไฟฟ้าที่ให้มา

## สิ่งที่เปลี่ยนจากเวอร์ชันก่อน

000000000000000. **(อัปเดตล่าสุด) เชื่อมข้อมูล Chiller Plant แบบ Live จริงแล้ว — ไม่ใช่แค่โค้ดตัวอย่างอีกต่อไป**

    ก่อนหน้านี้ฟังก์ชัน `getLiveChillerPlantData()` เขียนไว้แล้วแต่ยังไม่ได้ต่อเข้ากับ `doGet()` จริง (มีแค่
    ตัวอย่างโค้ดในคอมเมนต์) — **ตอนนี้ต่อเข้าใช้งานจริงแล้ว** พร้อม fallback อัตโนมัติ (ถ้าเชื่อมไม่ได้ ไม่ว่า
    เหตุผลอะไร จะไม่ทำให้ทั้งหน้าเว็บพัง แค่ใช้ค่าจำลองต่อไปเหมือนเดิม)

    **สิ่งที่ต้องทำเพิ่มอีกแค่ 1 ขั้นตอนก่อนเห็นข้อมูลจริง:**
    1. เปิดไฟล์ `Chiller_Plant_Township_Daily_Chiller_Efficiency_Analysis` ใน Google Sheets (ไฟล์จริงที่ใช้
       บันทึกข้อมูลอยู่ ไม่ใช่ไฟล์ตัวอย่าง) → คัดลอก Spreadsheet ID จาก URL
       (`https://docs.google.com/spreadsheets/d/`**ตรงนี้**`/edit`)
    2. เปิด `apps-script/Code.gs` → หาบรรทัด `const CHILLER_LOG_SPREADSHEET_ID = 'PUT_...'` → แทนที่ด้วย ID
       ที่คัดลอกมา → บันทึก + Deploy ใหม่ (Deploy → Manage deployments → แก้ deployment เดิม → Version: New)
    3. **สำคัญ:** บัญชี Google ที่ Deploy Apps Script ตัวนี้ ต้องมีสิทธิ์เปิดอ่านไฟล์ Chiller_Plant_Township
       ได้ด้วย (เป็นเจ้าของไฟล์เอง หรือถูกแชร์ให้ดูอย่างน้อย "Viewer") ไม่งั้นจะเชื่อมไม่สำเร็จ

    เมื่อตั้งค่าเสร็จ หน้า Chiller Plant จะดึงค่า **Chiller_Machines, SCHP_Units, CT_Units, และสรุปภาพรวม
    (kW, TR, ประสิทธิภาพ, CHWS/CHWR, Outdoor Air Temp)** จากแถวล่าสุดในชีต Log จริงโดยอัตโนมัติทุกครั้งที่
    รีเฟรชหน้าเว็บ — เพิ่มการ์ด **"Outdoor Air Temp"** ในแผง Efficiency Breakdown ด้วย (ดึงจากคอลัมน์
    `Record_Tadb(OAT)-C` ในชีต Log จริง)

    **ยังไม่ครอบคลุม (บอกตรงๆ):**
    - **PCHP_Units / CDP_Units รายเครื่อง** ยังเป็นค่าจำลองอยู่ เพราะชีต Log จริงมีแค่ยอดรวมทั้งกลุ่ม ไม่มี
      ข้อมูลแยกรายปั๊ม (ยอดรวม PCHP/CDP ในแผง Efficiency Breakdown เป็นค่าจริงแล้วนะครับ แค่รายเครื่องยังไม่ใช่)
    - **อุณหภูมิ AHU รายเครื่อง (SA/RA/SAG)** ยังเป็นค่าจำลองทั้งหมด เพราะยังไม่มีระบบเก็บข้อมูลจริงในส่วนนี้
      (รอเชื่อมกับ AHU Outsource ตามที่คุยกันไว้ก่อนหน้า) — เพิ่มแค่ **Outdoor Air Temp** (อุณหภูมิภายนอกอาคาร
      ภาพรวม) ที่มีอยู่จริงในชีต Log เข้ามาแทน เพื่อให้อย่างน้อยมีข้อมูลอุณหภูมิจริงบางส่วนแสดงผล
    - **Temp Phase I / Temp Phase II** ยังคงขึ้น "รอข้อมูล" เหมือนเดิม เพราะไม่มีในชีต Log จริง (ต้องรอทีมไปวัด
      ค่าตามแผนเดิม)

000000000000. **(อัปเดตล่าสุด) อ่านค่ามิเตอร์จากภาพถ่ายอัตโนมัติ — ลดขั้นตอนจดมือ**

    ตามที่ปรึกษาเรื่องช่างต้องถ่ายรูปแล้วพิมพ์ค่าลงเองอีกที (2 ขั้นตอน เสี่ยงพิมพ์ผิด) — เพิ่มฟังก์ชันใน
    `apps-script/Code.gs` ให้ **อ่านค่าตัวเลขจากภาพถ่ายมิเตอร์อัตโนมัติด้วย Claude Vision API** แล้วกรอกลง
    Google Sheet ให้เองทันที ไม่ต้องพิมพ์มือแล้ว

    **วิธีทำงาน:**
    1. ช่างถ่ายรูปผ่าน AppSheet ตามปกติ (ไม่ต้องเปลี่ยนขั้นตอนหน้างานเลย)
    2. สคริปต์ `processNewMeterPhotos()` ทำงานอัตโนมัติทุก 5-10 นาที (ตั้ง Time-driven trigger)
       สแกนหาแถวที่มีรูปแต่ยังไม่มีค่าตัวเลข
    3. ส่งรูปให้ Claude อ่านค่าที่แสดงบนมิเตอร์ แล้วกรอกลงคอลัมน์ที่ถูกต้องให้อัตโนมัติ
    4. **เทียบค่าที่อ่านได้กับค่าครั้งก่อนหน้า** ถ้าต่างกันเกิน 50% หรือ Claude ไม่มั่นใจ (confidence ไม่ใช่
       "high") จะ**ไม่บันทึกทับเงียบๆ** แต่ทำเครื่องหมายไว้ในคอลัมน์ `OCR_Review` ให้คนตรวจสอบก่อน — ป้องกัน
       เคสแบบ P1-CH-04 ที่เจอค่า TR ผิดปกติในรอบก่อนหน้านี้

    **ก่อนใช้งานจริงต้องทำ 3 อย่าง:**
    1. สมัคร Anthropic API ที่ [console.anthropic.com](https://console.anthropic.com) เติมเครดิต (ค่าใช้จ่าย
       ประมาณ 0.005–0.02 USD ต่อรูป — ปริมาณรูปต่อวันของทีมน้อย ค่าใช้จ่ายรวมต่อเดือนจึงต่ำมาก)
    2. เอา API Key ไปใส่ใน Apps Script: **Project Settings (⚙️) → Script Properties → Add script property**
       ชื่อ `CLAUDE_API_KEY` ค่าเป็น API Key ที่ได้
    3. แก้ `APPSHEET_FOLDER_ID` ในโค้ดให้ตรงกับโฟลเดอร์ Google Drive ที่ AppSheet เก็บรูปจริงของคุณ (ดูได้จาก
       AppSheet Editor → Data → ชื่อ Data source → โฟลเดอร์ที่ผูกไว้) และแก้ `METER_PHOTO_PAIRS` ให้ตรงกับ
       ชื่อคอลัมน์รูป/ค่าจริงในชีตของคุณ (ที่ใส่ไว้ตอนนี้เป็นตัวอย่างตามชีต Log ที่เคยส่งมาให้ดู)

    **ข้อควรรู้เรื่องความแม่นยำ:** อ่านค่าจากจอดิจิทัล (LCD/LED) แม่นยำสูงมาก ส่วนมิเตอร์เข็มอนาล็อกจะแม่นยำ
    น้อยกว่าเล็กน้อยขึ้นกับความชัดของภาพ **แนะนำให้ทดลองใช้คู่ขนานกับการจดมือสัก 1-2 สัปดาห์ก่อน** ค่อยเลิกใช้
    วิธีเดิมทั้งหมด

00000000. **(อัปเดตล่าสุด) แยก Chiller เป็น 2 ระบบ — Condenser Side กับ Evaporator Side ไม่ให้น้ำปนกัน**

    ตามที่ทักท้วงมาว่ากล่อง "Chiller System" เดิมรวมทั้ง 2 ระบบไว้ในกล่องเดียว ทำให้ดูเหมือนน้ำหล่อเย็น
    (Condenser) กับน้ำเย็น (Evaporator) มาบรรจบกัน ทั้งที่ในความเป็นจริงเป็นวงจรที่แยกจากกันเด็ดขาด (ถ่ายเท
    เฉพาะความร้อนผ่านสารทำความเย็นภายในเครื่อง ไม่มีน้ำไหลปนกัน) — แก้โดย:
    - แยกเป็น 2 กล่องชัดเจน: **"Chiller — Condenser Side"** (สีเขียวขอบ ต่อกับ Cooling System/CDP Pump)
      และ **"Chiller — Evaporator Side"** (สีฟ้าขอบ ต่อกับ PCHP/CH Supply Header) วางซ้อนกัน
    - เส้นประสีเทาบางๆ คั่นกลางระหว่าง 2 กล่อง พร้อมข้อความ "ถ่ายเทความร้อนเท่านั้น ไม่ใช่ท่อน้ำ" กำกับไว้
      ให้ชัดเจนว่าไม่ใช่ท่อเชื่อมน้ำ เป็นแค่การถ่ายเทความร้อนภายในตัวเครื่อง
    - เส้นท่อสีเขียว (Condenser) ต่อเข้าเฉพาะกล่องบน, เส้นท่อสีฟ้า/ส้ม (Evaporator/Primary) ต่อเข้าเฉพาะกล่องล่าง
      ไม่มีจุดใดที่เส้น 2 สีนี้ไปบรรจบกันที่จุดเดียวกันอีกต่อไป

    บอกทิศทางการไหล — เป็น "แผนผังตัวแทน" (Typical Diagram) ที่กำกับจำนวนเครื่องจริงและ kW จริงไว้ข้างๆ
    แต่ไม่ได้วาดครบทุกเครื่องทีละตัว (14 Chiller ฯลฯ) เพราะจะแน่นเกินอ่านง่าย — ดูรายละเอียดครบทุกเครื่องได้ที่

ในไฟล์นี้) แล้วต่อ Apps Script API แบบเดียวกับ Chiller Plant — สามารถใช้ Google Sheet คนละไฟล์กับ Chiller ได้

### 3. แผนเชื่อม IoT ในอนาคต (Flow / Temp In / Temp Out ทั้งฝั่ง Evap และ Cond)

โครงไว้ให้แล้วใน `apps-script/Code.gs` (ฟังก์ชัน `doPost` แบบ comment ไว้) แนวทางเมื่อซื้ออุปกรณ์จริง:

1. **เซนเซอร์ที่ต้องใช้ต่อ 1 Loop:** Flow meter 1 ตัว + Temp sensor ขาเข้า 1 ตัว + Temp sensor ขาออก 1 ตัว
   (รวม 2 Loop คือ Evaporator กับ Condenser = 6 จุดเซนเซอร์ต่อ Chiller Plant ชุดหลัก)
2. **สูตรคำนวณ kW จาก Flow + ΔT** (สูตรมาตรฐานระบบน้ำเย็น):
   `kW ≈ Flow (m³/h) × 1.163 × ΔT (°C)` (ค่าคงที่ 1.163 มาจาก density × specific heat ของน้ำ ที่หน่วย m³/h)
3. **อุปกรณ์ IoT ส่งค่ามาที่ Apps Script ผ่าน HTTP POST** (`doPost` ที่เตรียมโครงไว้) → เขียนต่อท้ายแท็บ
   `Chiller_Plant_Summary` โดยอัตโนมัติ แทนที่ค่าจำลอง (Demo) ที่ใช้อยู่ตอนนี้
4. เมื่อมีข้อมูลสะสมครบ 1 ปี ตัวเลข **kWh/m²/ปี** ในหมวด Area Efficiency จะเปลี่ยนจาก "ประมาณการ" เป็นค่าจริง
   (ตอนนี้คำนวณจาก Design_kW × ชั่วโมงสมมติ ปรับได้ที่ `ASSUMED_ANNUAL_OPERATING_HOURS` ใน `config.js`)
5. แนะนำเลือกอุปกรณ์ที่รองรับ Modbus RTU/TCP หรือ BACnet เพื่อให้ต่อผ่าน Bridge เดียวกับที่วางแผนไว้สำหรับ
   Trane CPMS ได้ (ดูคู่มือ `Trane_CPMS_Field_Guide.pdf` จากรอบก่อนหน้าประกอบ)
